Time and Memory Limits

Judger uses POSIX APIs to monitor your program. Notice that MEMORY LIMIT is not in KiB. We all know that 1 KB isn't 1024 bytes but 1000 bytes. We limit your program's resource usage accurately, so, just write your codes freely. For limits' details, see problem's page.

Input and Output

Pay ATTENTION that in NOI series you use file I/O but Standard Input/Output (aka "stdio") are used here. Judger will execute freopen automatically.
For example, you include <stdio.h> and use the function printf / scanf in C or you include <iostream> and use the template std::cin / std::cout in C++ for I/O.

Explanations for Frequently Occurred Issues

  • We improved the safety of OS Environment, dangerous header files or MACROs / functions are disallowed.
    For example, system in stdlib is disallowed.
  • In 64-bit environment, a pointer costs 8 bytes.
  • Function main must return a integer, 0 (zero), or causes Runtime Error.
  • The placeholder for a long long (64 bits) is %lld.
    But in some versions of Dev-C++, you use %I64d, or cause strange issues.
  • Function itoa is not available, use sprintf(charbuff, "%d", integer).
  • Pay attention to the differences between MS Windows' "\r\n" and Unix-Liked OS' "\n".
  • Make your program has no warnings if possible.

Why does my program run well on my computer, but fail on OJ?

  • Judger is based on Linux, and we use GCC. When comparing your output with standard output, Judger ignores spaces, line feeds and etc. by default.
  • Judger also limits your stack memory (10240KB). Judger throws a RUNTIME ERROR when stack overflows.
  • Every variable should be initialized. You may assume that a variable is 0 but in fact, it is not.
  • Linux has a strict control on memory access. Some invalid pointer may not cause fatal error on Windows but it cannot run on our OJ.
  • Serious memory leak issue may make your process killed by system protection module. So remember that you should release any memory you allocated.
  • Caution: binary float operations differ from decimal float operations.
  • If you learned over two types of programming languagse, you can try to 'translate' your code to another language and then submit. You may find some mistakes when translating.

常见问题解释

  • 我们加强了操作系统安全性，危险的头文件或宏或函数是禁用的。比如，stdlib头里的system是不被允许的。
  • 在64位环境中，一个指针占用8个字节。
  • main函数必须返回一个整数，0，否则会造成运行时错误。
  • long long类型的占位符是%lld。
    但有时在Dev-C++中需要使用%I64d，否则会造成奇怪的错误。
  • itoa函数不可用，使用sprintf(charbuff, "%d", integer)代替。
  • 注意微软视窗的"\r\n"和类Unix操作系统的"\n"之间的区别。
  • 尽量使得您的程序没有警告。

为什么程序在我的电脑上能够正常运行，而在评测系统上不能？

  • 评测系统建立在Linux下，编译器为GCC。评测系统在比较你的输出时默认采用忽略一切无效字符（空格，回车等）的策略。
  • 评测系统对你的程序内存的使用进行限制，同时也对你的程序栈的使用进行限制（10240KB）。如果你的程序使用递归多达100,000层或更多，那么你的程序很可能运行时出错。
  • 评测系统和你的电脑使用的内存安排方式可能不同。某些在你的电脑上没有经过初始化，理应为0的变量在评测系统上有可能并不如你所想的那样。
  • Linux对内存的访问控制更为严格，因此在Windows上可能正常运行的无效指针或数组下标访问越界，在评测系统上无法运行。
  • 严重的内存泄露的问题很可能会引起系统的保护模块杀死你的进程。因此，凡是使用动态内存分配而得的内存空间，请记得释放。
  • 注意浮点运算，二进制浮点数运算的时候很有可能会造成意想不到的差异。
  • 如果你会两种以上的语言，不妨将你的代码“翻译”成另一种语言然后提交，或许在翻译的时候你会发现你的程序的错误。如果翻译以后能够正常通过，那么请仔细检查你原来的程序。
